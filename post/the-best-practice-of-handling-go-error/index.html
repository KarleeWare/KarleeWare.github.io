The problem with the current error is twofold:

  • Can’t wrap more information, such as the call stack, such as layers of wrapped error messages
  • Can’t handle type information very well, e.g. I want to know if the error is of type io or type net

Wrap more information

There are many tools for this, the best known being https://github.com/pkg/errors, which has three main functions.

  • Wrap encapsulates the underlying error, adds more messages, and provides call stack information, which is missing in native error
  • WithMessage encapsulates the underlying error, adds more messages, but does not provide call stack information
  • Cause returns the bottom error, stripping away the layers of wrap

// Wrap returns an error annotating err with a stack trace
// at the point Wrap is called, and the supplied message.
// If err is nil, Wrap returns nil.
func Wrap(err error, message string) error {
	if err == nil {
		return nil
	}
	err = &withMessage{
		cause: err,
		msg:   message,
	}
	return &withStack{
		err,
		callers(),
	}
}

type withStack struct {
	error
	*stack
}

func (w *withStack) Cause() error { return w.error }

func Cause(err error) error {
	type causer interface {
		Cause() error
	}

	for err != nil {
		cause, ok := err.(causer)
		if !ok {
			break
		}
		err = cause.Cause()
	}
	return err
}

Cause recursive calls that do not implement the causer interface return this err

func Errorf(format string, a ...interface{}) error {
	p := newPrinter()
	p.wrapErrs = true
	p.doPrintf(format, a)
	s := string(p.buf)
	var err error
	if p.wrappedErr == nil {
		err = errors.New(s)
	} else {
		err = &wrapError{s, p.wrappedErr}
	}
	p.free()
	return err
}

func (p *pp) handleMethods(verb rune) (handled bool) {
	if p.erroring {
		return
	}
	if verb == 'w' {

		// or with a non-error arg.

			return true

func Is(err, target error) bool {
	if target == nil {
		return err == target
	}

	isComparable := reflectlite.TypeOf(target).Comparable()
	
	for {
		if isComparable && err == target {
			return true
		}
		if x, ok := err.(interface{ Is(error) bool }); ok && x.Is(target) {
			return true
		}
		if err = Unwrap(err); err == nil {
			return false
		}
	}
}

Is function is relatively simple, recursive layer by layer check, if it is nested err, then call Unwrap layer by layer to find the bottom err, and finally determine whether the pointers are equal

var errorType = reflectlite.TypeOf((*error)(nil)).Elem()

func As(err error, target interface{}) bool {
	if target == nil {
		panic("errors: target cannot be nil")
	}
	val := reflectlite.ValueOf(target)
	typ := val.Type()
	
	if typ.Kind() != reflectlite.Ptr || val.IsNil() {
		panic("errors: target must be a non-nil pointer")
	}
	
	if e := typ.Elem(); e.Kind() != reflectlite.Interface && !e.Implements(errorType) {
		panic("errors: *target must be interface or implement error")
	}
	targetType := typ.Elem()
	for err != nil {
		if reflectlite.TypeOf(err).AssignableTo(targetType) {
			val.Elem().Set(reflectlite.ValueOf(err))
			return true
		}
		if x, ok := err.(interface{ As(interface{}) bool }); ok && x.As(target) {
			return true
		}
		err = Unwrap(err)
	}
	return false
}

It also recursively calls As, while Unwrap the bottom error, and then uses reflection to determine if the assignment is possible, and if it is, then it is of the same type

func ExampleGroup_justErrors() {
	g := new(errgroup.Group)
	var urls = []string{
		"http://www.golang.org/",
		"http://www.google.com/",
		"http://www.somestupidname.com/",
	}
	for _, url := range urls {
		// Launch a goroutine to fetch the URL.
		url := url // https://golang.org/doc/faq#closures_and_goroutines
		g.Go(func() error {
			// Fetch the URL.
			resp, err := http.Get(url)
			if err == nil {
				resp.Body.Close()
			}
			return err
		})
	}
	// Wait for all HTTP fetches to complete.
	if err := g.Wait(); err == nil {
		fmt.Println("Successfully fetched all URLs.")
	}
}

Above is the official example given, the underlying use of context to cancel other requests, synchronization using WaitGroup, the principle is very simple, the amount of code is very small, interested in the source code can see

Three points must be noted here:

  • Who passed in the context? Other code will not be used, cancel can only be executed once
  • g.Go does not come with recover, for the robustness of the program, you must recover
  • parallel goroutine returns with an error instead of collecting results after a normal fan-out request

A few issues to note in online practice

error and panic

If you look at the go source code, you will find that there are many places to write panic in the source code, but in engineering practice, especially the business code should not actively write panic Theoretically, panic only exists in the server startup phase, such as config file parsing failure, port listening failure, etc. All business logic is prohibited to actively panic.

According to the CAP theory, the most important thing on the web Internet is AP, high availability is the most critical (non-banking financial scenario), if there is a part of the word list, metadata loading failure when the program starts, can not panic, to provide services is the most critical, of course, to have an alarm, so that the development of the first time to perceive the current service QOS has been reduced. All asynchronous goroutine are to use recover to underwrite the processing

So here is a guideline: data transfer and exit control, you need to use a separate channel can not be mixed, we generally use context to cancel the asynchronous goroutine, rather than directly close channels
