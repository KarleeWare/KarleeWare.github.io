A few must-have Golang tricks

2022-04-11

Go is a very good programming language. It is a language that allows you to really focus on your business and not have to worry too much about the program itself, so you can write applications as quickly as possible. For example, it has a relatively complete ecosystem that can provide you with everything you need to get started.

But then, it’s not a panacea, and there are some things we need to be aware of. I’m writing this article mainly as a reminder to myself, and also as some observations and summaries. Of course, these are just tips, not real problems, but of course if you notice them and use them in the real world, then it will benefit you a lot, which is the difference between a newbie and a veteran.

Do not use Logrus

This is actually related to generics. Because Go is a strongly typed static language, you can’t bypass data types like you can with NodeJS or PHP. So what if we still need to use generic types? Like Loger, for example, or ORM, because the only way to write generic code is to use generic types, otherwise you’d have to write each one once.

In the end, we have to use reflection. And Logrus makes heavy use of reflection, which leads to a lot of allocation counts. While not usually a big problem (depending on the code), performance is important, especially in large scale, highly concurrent projects. While this may sound like a very minor optimization, it is important to avoid reflections. If you see code that can use structure regardless of type, it will use reflection and have a performance impact.

For example, Logrus doesn’t care about types, but obviously Go needs to know (eventually).What does Logrus do? Use reflection to detect types, which is overhead.

  log.WithFields(log.Fields{
    "animal": myWhatever,
  }).Info("A walrus appears")

So I would prefer zerolog, but of course zap is good too. Both claim zero allocation, which is what we want, since they have the least performance impact.

Do not use encoding/json

When we need a function, function, many people suggest using the standard library. The exception is the encoding/json module in the standard library. In fact, like the example above, encoding/json uses reflection, which can lead to poor performance and losses when writing APIs that return json responses, or microservices.

You can use Easyjson, for example, which is simple and efficient, and uses a code generator to create the code needed to convert the structure to json to minimize distribution. This is a manual build step and is annoying. Interestingly json-iterator also uses reflection, but is significantly faster and I suspect black magic.

Do not use closures in goroutines if possible

For example, the following sample code

for i:=0;i<10;i++ {
  go func() {
     fmt.Println(i)
  }()
}

Most people probably expect this to print the numbers 0 through 9, just like when delegating the task to goroutine.

But the actual result: depending on the system, you will get one or two numbers and many 10s.

Why is this? Closures have access to the parent scope, so you can use variables directly. Although updated linters may warn you about “variable closure capture”, they do not require you to redeclare the variable.

Go’s reputation for performance owes much to the execution of runtime optimizations, which try to “guess” what you want to do and optimize certain execution paths. During this time, it “captures” variables and passes them to where they are needed in the most theoretically efficient way possible (for example, after completing some non-concurrent operation to free up some CPU allocations). The result in this case is that the loop may start goroutines, and the goroutines may receive the value of i from the parent scope much later. There is no guarantee which one you will see on multiple executions of this code, it could be the number 10 or some other number.

If you do use a closure for some reason, be sure to pass the variable i and treat the closure as you would every function.

Summary

There are definitely more than these three techniques in programming practice. Figuring them out and mastering them in practice will improve our programming skills and allow us to write more optimized code.
