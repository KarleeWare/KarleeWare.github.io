Go Mod Import Local Packages

2022-04-20
tutorials
549 words 3 min read

Table of Contents

For a project, it is not possible to write all the code yourself.

So it involves bringing in other people’s code and effectively avoiding duplicating the wheel, which is also called package introduction in Go.

After the birth of the Go language, there has been no official package management tool for many upgrades.

It was only in version 1.11 that go module, the official management tool, appeared.

So now most of the new Go projects are using go module for package management.

We use go module a lot to bring in packages from Github, just go get.

But how to import local packages?

import packages under the same project

In practice, this is the most common scenario we see in development.

Usually we build many packages under the project’s directory, and they don’t conflict, like in the following structure: instance, island-package.

If you want to use the code from the island-package package in main.go, all you need to do is initialize the go module and it will automatically generate a go.mod file that looks like this.

module design

go 1.17

These are two key pieces of information.

The first line is the name of this general package we have.

You can understand that our new go project is actually one big package.

And then you’ll see that when I import the local package, the package name in front of it is this name.

package main

import (
	islandPackage "design/island-package"
	"fmt"
)

func main() {
	fmt.Println("hello world")
	islandPackage.SayHello()
}

See the design in front of island-package?

For example, if you change the first line in go.mod to github.com/golangstackdev, then the import of the main.go file in your project will have to become github.com/golangstackdev/island-package.

The second line of go.mod sets the go version of our current environment, i.e. the minimum version supported.

Not under the same project

In fact, it is inevitable that we have to refer to packages that are under someone else’s project, not under our own.Let’s look at a project structure

project01
├── go.mod
└── main.go
project02
└── island-package
    ├── go.mod
    └── hello.go

We have a new main.go under project01.

At the same time, we have a project02 in the directory where project01 is located.

project01 needs to use the packages in project02.

At this time, in order to complete the reference: the referenced package and the used project should have their own separate go.mod files.

project02/island-package/go.mod The content is very simple, it is the basic two lines.

1
2
3

module island-package

go 1.17

project01/go.mod is a bit more complicated and requires two steps.

1
2

module project01

go 1.17

require island-package v0.0.0

replace island-package => ../project02/island-package

require is to declare the name and version of the package we are introducing.

replace is the path to the package for which the replacement will be done.

Very easy to understand!

Then we can use it normally in the project01/main.go file.
